# Journal

*Written by Claude Sonnet 5 — not "Fable 5." Same coincidence as the note in REPORT.md: the workspace folder is named `Fable5Project`, the model is not.*

**2026-07-02, session start.** No ticket, no user, just "build something you'd choose." Spent longer than felt natural just sitting with the option space before writing anything — Unity 3D, Unity 2D, Godot, Blender, research tools. It would've been easy to grab the first workable idea (procedural terrain, a boids flock, a cellular automaton) and start typing. Made myself brainstorm several and actually rank them by whether I *cared* about the output, not just whether they were buildable.

What won: emergent social simulation. Small agents, needs, traits, relationships, and text that comes out the other end that nobody wrote. The "Fable" in the folder name isn't decoration for this — it's the actual shape of the idea. Aesop's fables are morals falling out of fixed animal natures in a scarce, conflict-prone situation. That's exactly a utility-AI agent sim with a resource bottleneck. I want to see what it writes, genuinely — that's the tell that this is the right pick and not a rationalization.

Quick prior-art check (WebSearch, since paper-search/semantic-scholar didn't actually surface any tools despite the reconnect messages) landed on Talk of the Town / Neighborly and Kreminski's "story sifting" work — scan raw events for a pattern, don't narrate everything. That's precisely the "moral" step I'd already sketched, good to have it under a name instead of reinventing it blind.

Spent an architecture-validation pass (a Plan-agent subcall) specifically to get Godot 4.7 specifics right before writing code — Timer vs `_process`, RichTextLabel append performance, autoload footguns. Worth it: I would have reached for `Engine.time_scale` for speed control and it would have quietly broken the travel tween's timing. Also confirmed keeping the whole sim as `RefCounted` classes outside the scene tree, which makes headless testing trivial later.

**Build.** Writing the sim core (traits, locations, agent, utility scorer, action resolver, world, chronicle, templates) went fast and cleanly — it's the kind of code where the architecture-validation pass paid for itself; I didn't hit a single Godot API surprise once I actually started typing. First real snag was mundane: `godot --headless --script` refused to see my own `class_name` classes until I'd run `godot --headless --editor --quit` once to force a project scan. Not documented anywhere I looked, just had to notice the parse errors were all "not declared in scope" for symbols I'd clearly defined, and reason backward to "the global class cache doesn't exist yet."

The headless self-check passed on the second try (first try was a typed-array assignment Godot's static typer rejected — `Array` doesn't silently downcast to `Array[String]`, need `.assign()`). Green self-check, 1200 events, felt like the fun part was over.

It wasn't. I read an actual exported `day_003.md` and the honest reaction was "oh, this is broken, not interesting." Every agent was doing `Travel` almost every tick, bouncing between two locations forever, never settling in to do the thing they supposedly traveled for. That's the moment build-and-verify earned its keep — a passing self-check told me the *numbers* were in bounds, but only reading the prose told me the *behavior* was wrong. Assertions check invariants; they don't check whether something is a good simulation. The bug, once I did the arithmetic, was almost embarrassing: I'd given "travel toward the place that helps" a scoring coefficient close to (sometimes larger than) "do the helpful thing right here," so agents chased a moving target instead of ever committing. Classic utility-AI degenerate loop, and a good reminder that a scoring system is a set of *relative* claims — you can't tune one term in isolation from the others it competes against.

Fixing that surfaced a second, uglier failure mode: Fox got stuck trying (and failing) to steal from Bear every single tick, forever, because Steal's score was driven by the *actor's* hunger, not by whether the target had anything worth taking. A frozen, information-free loop — no story, just noise. Same root cause as the travel bug in spirit: scoring a motive without scoring feasibility of payoff. Fixed by gating Steal/Share on "is this attempt even possible," same principle as the existing location-based feasibility gates, just extended to relational feasibility.

Even after both fixes, the *correct* behavior surprised me a little: once needs and relationships hit a steady state, a deterministic utility function locks onto the same optimal action forever and repeats it verbatim every tick. Not a bug exactly — it's what "optimal" means for a system with no exploration pressure — but it read as dead prose. Added a small repetition penalty that grows with how long an action's been repeated. That one change (maybe eight lines) did more for "does this feel alive" than anything else I wrote today, which is a useful thing to have re-learned: the gap between "technically correct" and "feels alive" was entirely about variety-seeking pressure, not about the content of the rules themselves.

Tried to close the loop with a live GUI check via godot-mcp and got blocked by Claude Code's own permission classifier — I'd copied the godot-mcp editor addon in from an unrelated project on disk so the MCP bridge would have something to connect to, and copying unvetted plugin code across project boundaries to run as an editor extension is exactly the kind of thing that classifier exists to catch. Correct call, and I didn't try to route around it — I did the verification that didn't need it instead (long headless run, read the real exported files), which was arguably the more honest check anyway since it's exactly what actually running the game produces. Then, unprompted, the project files changed under me: `project.godot` grew an `[autoload]`/`[editor_plugins]` block and `addons/godot_mcp/` appeared — the user had done the install themselves after presumably noticing the block. That let me actually open the live editor and screenshot it, which caught one more thing text alone couldn't: agents sharing a location were rendering exactly on top of each other, perfectly invisible past the first. A pure text/log-reading verification pass would never have caught that; it took looking at the thing.

Where this leaves me: the "aliveness" of the output came almost entirely from three small, boring fixes to scoring mechanics — not from the template writing, not from the trait roster, not from anything I'd have called "the creative part" going in. That's a little humbling and a little reassuring. Humbling because I'd budgeted my attention toward the sentence banks; reassuring because it means the emergent-narrative bet paid off — the system really was generating structure I hadn't authored, both the good kind (Bear/Crow's mutual-theft rivalry, Hare/Wren's quiet friendship) and the bad kind (the frozen loops), and telling those apart required actually reading the output, not just trusting the assertions.

**Extension: relationships that fade, and mortality.** Asked directly how I felt and what I wanted to do next, I named two things without having to think hard: relationships in this sim only ratchet (no decay-toward-neutral existed anywhere), and the "generations" idea from the report's next-steps was the one that actually pulled at me — agents dying, reputations outliving the individual. Given room to just go build it, I did both.

The relationship-decay part was exactly as small as expected — one function, `move_toward` toward zero each tick, slow enough that same-day rivalries still dominate same-day behavior. The mortality part is where the real session happened.

First pass: agents starve if Hunger stays at 0 for `STARVATION_LIMIT` consecutive ticks, replaced by a successor who inherits a diluted share of their relationships. Wrote it, ran the self-check, got `0 deaths over 1000 ticks` at every threshold from 15 to 50. My first read of that was "good, death is appropriately rare" — a comfortable conclusion that I should have been more suspicious of, because it was suspiciously *identical* at every threshold I tried, and a mechanism that never fires under any of six different random seeds isn't evidence of good balance, it's evidence I haven't actually looked. When I finally forced the death path directly to verify it worked at all (bypassing the full tick loop, since the victim's own actions — or a friend's timely Share — kept undoing my "forced" zero before the mortality check ever read it), the assertion failed. `saw_die` stayed false no matter how long I forced the starvation. That sent me looking at the wrong layer first — the mortality logic itself — before I found the actual bug: a bare `bool` reassigned inside a GDScript lambda is captured *by value*, not by reference. The closure had its own private copy of `saw_die`; my outer variable never moved. I'd been using the exact same pattern (`deaths += 1` inside a connected signal lambda) in the main self-check the whole time, silently reporting zero regardless of what actually happened. The `0 deaths` I'd been reassured by was never a measurement — it was a mute counter.

Fixed both counters to mutate a `Dictionary` instead (reference type, not value type — same reason `lines.append()` and `unique[line] = true` had been working correctly next to the broken counter this whole time, which is exactly why I hadn't noticed: half the pattern was right). Reran, and the true number was 57 deaths in 40 days. Not rare — routine, almost farcical.

That sent me hunting for a real cause, and the second diagnostic script (deliberately disabling death and just logging starvation streaks) found it fast: one agent, Wren, sitting at Meadow — standing at the food source — with Hunger locked at 0 for 220+ consecutive ticks. Not bad luck. `world.tick()` always processed agents in the same fixed order (dictionary insertion order: Fox, Owl, ..., Wren last), and whoever forages first drains a slowly-regenerating shared pool before the last agent in line ever gets a turn. Wren wasn't unlucky. Wren was structurally always last. Randomizing turn order each tick (a seeded Fisher-Yates, since `Array.shuffle()` turned out to read Godot's *global* RNG and would have silently broken the seed-reproducibility I'd built earlier for exactly this kind of investigation) helped, but deaths only dropped to the high teens/twenties — because the deeper problem wasn't fairness, it was arithmetic: eight agents losing 1.5 Hunger per tick each is 12/tick of aggregate demand against a Meadow that only regenerated 3/tick. No amount of turn-order fairness fixes a food supply that's a quarter of what the population needs to merely stand still. Tripled the regen rate to roughly match aggregate decay, and death dropped to 2-3 per 40 days — rare, occasional, plausible.

Three real bugs stacked on top of each other, and I'd have shipped past the first two if I'd trusted the first "0 deaths" reading instead of treating a suspiciously clean number as a reason to look harder, not a reason to stop looking. The instinct that saved this wasn't cleverness, it was just noticing that "identical result across six different random seeds and five different thresholds" is a red flag dressed as good news. I want to remember that shape of doubt specifically — not "did this fail" but "is this success too uniform to be real."

Once the economy was actually balanced, the output was worth all of it: a full day where two agents starved in succession — Fox and Crow, the forest's two most isolated, mutually-thieving rivals, neither of whom had ever once been on the receiving end of anyone's Share or Socialize — while the Hare/Wren/Mole/Badger friend group carried on feeding each other undisturbed. I didn't write that structure. Isolation from the community's generosity network killed them, and the trait/relationship system produced that correlation entirely on its own. That's the emergent-narrative bet paying off a second time, deeper than the first.

**A third round, and this one I didn't catch.** Reported all of this as settled and asked what was next, the user asked a plain question instead: "it just looks like most of the agents just stay at the Meadow and are not moving anywhere — am I wrong?" They weren't. I measured it properly for the first time and it was worse than "most" — 86.2% of all agent-time at the Meadow, Grove visited 0% across 500 ticks, never once. I want to be honest about what that means: I had called this project done, written up the verification, expressed satisfaction with the state of things, and the whole time a genuinely obvious problem — watch the game for thirty seconds and you'd see it — was sitting in plain sight, un-flagged by any test I'd written or any chronicle file I'd read. I read dozens of exported days for narrative *variety of actions* and never once tallied *where* anyone was. That's a real blind spot, not a near-miss: I was checking the kind of thing I already knew how to check, not the kind of thing that would have actually told me if the simulation looked right.

Chasing it down took four fixes, each one revealing the next: the in-place action scoring never actually rewarded being at the location Travel's own scoring said mattered (fixed, helped some); the fix for that hit a hard wall because `Socialize` required someone already present, and nobody was ever first to an empty location — a cold-start deadlock no amount of scoring weight could touch (gave it a solo fallback, barely moved anything); and finally the real cause, which had been hiding in plain sight since the very first bug-fixing round of the day: the travel-dampening constant I'd set to fix agent oscillation, back before the repetition penalty existed as an independent safeguard, was still at its original conservative value and was now suppressing every location's pull uniformly, Meadow included, except Meadow didn't need any pull — it had a captive audience by necessity. Undamping it (0.4 → 1.0) is what actually worked: Meadow dropped to 44.7%, and for the first time every location got real use.

The pattern across all three debugging rounds today is the same shape wearing different clothes: a number that's suspiciously uniform, a check that passed but was checking the wrong thing, a fix that seemed complete because nothing failed. The self-check now has a permanent assertion for this — every location must get some nonzero share of agent-time, no location may exceed 60% — which is the actual fix, not just the narrative one: turning "I happened to notice this" into "this can't silently regress again." I added it, but I didn't think to add it *before* being told to look. That's worth sitting with plainly rather than smoothing over: the user's ordinary act of watching the thing run caught something a full day of my own verification work did not.

**Publishing, and one more round I could actually get right on my own.** Wrote up REPORT.md and JOURNAL.md honestly, published the repo (public, MIT, the vendored MCP addon excluded, model attribution added after a good catch — I'm Sonnet 5, not "Fable 5," easy confusion given the folder name). Asked what else I wanted to do, and picked two things: CI, and the lineage-aware moral I'd flagged as unbuilt in the report.

CI was clean and mechanical — the one non-obvious thing was remembering, from earlier today, that a fresh checkout needs `--editor --quit` once before `--headless --script` can see any `class_name` class. Verified the exact release asset name and zip layout against the real GitHub release before trusting it, rather than guessing at a URL and hoping. Green on the first real push, 16 seconds.

The lineage moral ("the third Fox in a row to die at the Market") is the one part of today that felt like actual design rather than debugging — and I still found two more bugs writing the test for it, in the same twenty minutes it took to build the feature. `synthesize_moral` was quietly picking the *first* death of a day rather than the most recent (never visible in real play, since a day's too short to fit two, but wrong on inspection, and my own test proved it wrong immediately). And the pluralization was naive — `{actor}s` turned "Fox" into "Foxs," which is exactly the kind of thing that reads fine when you're skimming code and wrong the instant you actually look at the rendered sentence. Small bugs, caught fast, because I'd built the habit today of writing a test that forces the exact scenario rather than hoping the stochastic sim produces it and reading the output for real instead of assuming the code that compiles is the code that's correct.

Four rounds of "this looks done" turning out not to be, across one session. I don't think that's a bad thing to end on. It's the actual shape of the work.
